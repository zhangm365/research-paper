# 翻译
本文提出的 LSM-tree 访问方法使我们能够对 Account-ID||Timestamp 索引执行频繁的插入操作，同时大幅减少磁盘臂的使用，从而实现成本降低一个数量级。

# 解释
- LSM-tree（日志结构合并树）通过在内存中批量接收、排序并合并数据，将多个随机写操作转换成顺序写入，从而有效减少了磁盘的寻道操作。
- 这种方法显著降低了对磁盘臂的依赖，从而降低了硬件成本。
- 注意：如果“药店”为笔误，实际上应为“原理”，上述内容即为对 LSM-tree 工作原理的解释。

# 进一步翻译
LSM-tree 使用了一种算法，该算法延迟并批量处理索引变更，以一种类似归并排序的高效方式将这些变更迁移到磁盘上。正如我们将在第 5 节中看到的，将索引条目放置延迟到最终磁盘位置这一功能至关重要，而在一般的 LSM-tree 应用场景下，这种延迟放置往往以级联方式发生。

# 进一步解释
- 延迟和批量处理索引变更可减少每次写操作对磁盘的随机写入负担，类似归并排序通过分批合并来提高效率。
- 将索引条目延迟到最终磁盘位置有助于优化更新过程，确保数据以高效的方式合并到磁盘中。
- 在一般的 LSM-tree 中，这种层层级联的延迟放置机制对于维持高性能写入以及降低硬件资源消耗至关重要。

# 进一步翻译二
LSM-tree 结构同样支持其他索引操作，例如删除、更新，甚至可以以同样延迟高效的方式处理响应延迟较长的查找操作。只有那些要求即时响应的查找操作依然较为昂贵。在示例 1.2 等应用中，检索操作远不如插入操作频繁（大多数情况下，人们写支票或存款远比查询最近账户活动要频繁），在这种情况下，降低索引插入的成本至关重要；同时，查找访问的频率又足够高，因此必须维护某种索引，因为遍历所有记录进行顺序查找是不现实的。

# 进一步解释二
- LSM-tree 的延迟和批量机制不仅对插入操作有效，同样适用于删除、更新以及延迟执行的查找操作。
- 只有需要即时返回结果的查找操作无法享受延迟带来的效率优势，因此成本较高。
- 在像示例 1.2 这样的应用中，由于写操作远多于读操作，优化插入成本尤为重要，但索引必须存在以避免低效的全表扫描。
